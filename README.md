# Simple Point API (feat. concurrency problem)
## 소개
본 프로젝트는 간단한 **포인트 조회/사용 API**를 모델로, 멀티 스레드 환경에서 발생할 수 있는 **동시성 문제**를 해결하는 과정을 담은 토이 프로젝트입니다. 이 프로젝트의 베이스 코드는 *항해 플러스 백엔드 7기 1주차 스켈레톤 코드*을 활용합니다.

[API 명세서 이동](/docs/api-spec.md)

### 주요 목적
**순수 Java의 동시성 제어 도구**만 이용하여 동시성 문제를 해결하는 데 중점을 둡니다.

동시성 제어의 기초를 학습하기 위한 목적으로 내부 아키텍처는 의도적으로 단순하게 구성되었습니다.

### 제약 사항
- 별도의 DB 연결 없이, `In-Memory` 데이터 저장소(`List`, `Map` 등) 사용을 가정한다.
- 분산 환경은 고려하지 않는다. (단일 서버 환경을 전제로 한다.)

### 요구 사항
- 다수의 스레드가 동일한 사용자 포인트에 동시에 접근하는 경쟁 상태(`Race Condition`) 해결
    - 동시에 들어오는 여러 건의 포인트 충전, 이용 요청은 순차적으로 (혹은 한 번에 하나의 요청씩) 처리되어야 한다.
- 동시성 제어에 대한 통합 테스트 작성

## 동시성 문제 분석
### 동시성 문제
다수의 프로세스나 스레드가 하나의 자원(e.g. DB, 파일, 메모리 등)에 동시에 접근하여 쓰기 작업을 수행할 때 데이터의 일관성, 정합성이 깨지는 문제를 말한다.

대표적인 증상은 다음과 같다.

- Race condition: 두 스레드가 같은 값을 읽고 각자 수정 → 한 쪽 갱신 유실 (Lost Update)
- Dirty read: 아직 커밋되지 않은 데이터를 다른 트랜잭션이 읽음
- Non-repeatable read: 같은 트랜잭션 내 동일 쿼리 결과가 달라짐

### 시스템 환경에 따른 동시성 제어 기법
1. 단일 서버 환경
    - 스레드 동기화 (Application 레벨 / Java)
        - `synchronized`
            - 메서드나 코드 블록에 사용되어 한 번에 하나의 스레드만 진입할 수 있도록 한다.
        - `ReentrantLock`
            - `synchronized`보다 더 정교하게 락을 제어할 수 있다.
            - 공정 옵션과 같은 고급 동기화 기능을 제공한다.
            - 재진입이 가능하며, `lock()`, `unlock()` 메서드를 통해 락을 명시적으로 제어할 수 있다.
            - `tryLock()` 같은 기능을 통해 락을 획득하지 못했을 때 다른 작업을 수행하는 등 로직을 더 유연하게 구현할 수 있다.
        - `Atomic` 변수
            - 멀티스레드 환경에서 원자성을 보장해 주는 변수로, 내부적으로 **CAS(Compare-And-Swap)** 알고리즘을 사용하여 잠금 없이(lock-free) 원자적 연산을 보장한다.
            - Compare-And-Swap(CAS) 란?
                - 현재 스레드가 가지고 있는 기존 값과 메모리가 가지고 있는 값을 비교해 같은 경우 변경할 값을 메모리에 반영하고, 다른 경우에는 변경 값이 반영되지 않으며 재시도한다.
                - 락을 사용하지 않기 때문에 값이 업데이트되는 도중에도 다른 스레드의 동시 접근을 허용한다. (높은 동시성 제공)
        - `ConcurrentHashMap` 컬렉션
            - Lock, CAS 알고리즘, 노드 동기화 등의 기법으로 동시성을 지원하는 `HashMap` 자료 구조
            - 특정 키에 대한 동작만 락을 걸어 높은 성능을 유지
            - 읽기 작업은 락 없이 가능하며, 쓰기 작업은 세분화된 락으로 동기화한다.
    - 데이터베이스 락
        - 데이터베이스의 Row Level Lock 사용 (Pessimistic / Optimistic Locking)
        - Optimistic Lock
            - 충돌이 드물 것이라 가정하고, 잠금 없이 작업을 진행한다. 실제 충돌 시에만 처리하는 방식
            - 테이블에 버전(Version) 또는 타임스탬프(Timestamp) 칼럼이 추가되며, 데이터를 수정하기 직전에 버전이나 타임스탬프를 비교하여 충돌 여부를 확인한다.
        - Pessimistic Lock
            - 충돌이 자주 발생할 것이라 예상하고, 데이터를 읽는 순간 바로 락을 건다. 다른 트랜잭션은 잠금이 해제될 때까지 기다려야 하므로, 동시성은 낮지만 데이터의 무결성을 확실하게 보장한다.

2. 분산 서버 환경
    - 데이터베이스 락
        - 데이터베이스의 Row Level Lock 사용 (Pessimistic / Optimistic Locking), 위와 동일
    - 분산 락
        - Redis의 `SETNX`, ZooKeeper의 `EPHEMERAL` (임시) 노드를 사용하여 락을 구현
    - 메시지 큐
        - 동시성 문제를 직접 해결하기보다는 동시 발생 요청을 순차 처리함으로써 간접적으로 해결하는 비동기 방식
        - `RabbitMQ`, `Kafka`, `AWS SQS` 등


### 동시성 문제 발생 지점
#### 포인트 충전/사용
- 포인트를 동시에 충전/사용하는 경우, 포인트가 잘못 계산되어 데이터의 무결성이 깨질 위험이 있다. 대표적인 케이스로, 여러 트랜잭션이 동일한 데이터를 동시에 읽고 그 값을 기반으로 계산한 뒤 다시 쓸 때 동시성 문제가 발생한다. 포인트 시스템에서는 금전적 손실로 직결될 수 있다.

#### 문제 발생 시나리오
사용자 A는 10,000 포인트를 가지고 있다. 동시에 7,000 포인트를 사용하는 **요청 1**과 5,000 포인트를 사용하는 **요청 2**를 받았다고 가정하자

서버는 두 요청을 거의 동시에 받아 두 개의 스레드에서 각각 처리할 것이다.

| 시간 | 요청 1 (스레드 1) | 요청 2 (스레드 2) | DB |
| --- | --- | --- | --- |
| `T1` | **[조회]** 현재 포인트 조회 | - | 10,000|
| `T2` | (로직 처리 중) | **[조회]** 현재 포인트 조회 | 10,000 |
| `T3` | (로직 처리 중) | (로직 처리 중) | 10,000 |
| `T4` | **[계산]** 10,000 - 7,000 = 3,000 | (로직 처리 중) | 10,000 |
| `T5` |   | **[계산]** 10,000 - 5,000 = 5,000 | 10,000 |
| `T6` | **[갱신]** 포인트를 3,000점으로 갱신 |  | 3,000 |
| `T7` | (트랜잭션 종료) | **[갱신]** 포인트를 5,000점으로 갱신 | 5,000 |
| `T8` |   | (트랜잭션 종료) | 5,000 |

- 정상 결과
    - 사용자는 총 12,000점을 사용하려 한다.
    - 포인트가 10,000점뿐이므로 둘 중 하나의 요청은 '포인트 부족' 으로 실패해야 했다.
- 실제 결과
    - 두 요청 모두 성공으로 처리되었고, 요청 2가 `T7` 시점에 5,000점으로 덮어씀에 따라 요청 1의 갱신 내역 3,000점은 분실(`Lost Update`)되었다.

이러한 문제는 여러 스레드가 동시에 공유 자원에 접근하면서 발생하는 **경쟁 상태(`Race Condition`)** 때문이다.

> 💡 경쟁 상태(`Race Condition`)란?<br/>여러 스레드가 공유 자원에 동시에 접근하여 작업할 때 **실행 순서**에 따라 결과가 예상과 다르게(비정상적으로) 나타나는 상황을 말한다.

### 해결 방법
**[ReentrantLock: 공정 모드 지원]**<br/>
이 프로젝트는 DB 대신 자바 컬렉션 `List`를 데이터 저장소로 활용하므로 DB 레벨의 낙관적 락과 비관적 락, `Atomic`은 적용할 수 없다.

대표적인 두 가지 선택지인 `synchronized`와 `ReentrantLock`은 모두 상호 배제(`Mutual Exclusion`)(한 번에 하나의 스레드만 접근) 기능은 동일하게 제공한다.

하지만 이 프로젝트의 핵심 요구 사항인 **요청의 순차적 처리**를 보장하는 데는 결정적인 차이가 있다.

`synchronized` 키워드는 `ReentrantLock`과 마찬가지로 한 번에 하나의 스레드만 자원에 접근할 수 있도록 동시성을 제어할 수 있다. 그러나 락 획득의 순서의 공정성을 보장할 수 없어 요청이 순차적으로 처리될 수 없다.

- `synchronized` (비공정 락 - Non-Fair Lock)
    - `synchronized`는 락을 획득하는 순서를 보장하지 않는다.
    - 락이 해제되는 순간, 대기 중이던 스레드 중 하나가 경쟁을 통해 락을 획득한다.
    - 이 방식은 요청이 도착한 순서대로 처리하도록 보장하지 않으므로, 특정 스레드가 계속 락을 얻지 못하는 **'기아 상태(Starvation)'**를 유발할 수 있다.

- `ReentrantLock` (공정 모드 - Fair Lock)
    - 생성자(`new ReentrantLock(true)`)를 통해 **'공정 모드'**를 명시적으로 활성화할 수 있다.
    - 이 모드는 락을 획득하기 위해 대기하는 스레드들을 큐(Queue)로 관리하며, 가장 오래 기다린 스레드에게 락을 우선적으로 부여한다. (FIFO와 유사)

따라서, 요청이 들어온 순서대로 공정하게 처리하는 순차성을 만족시키기 위해 `ReentrantLock`을 채택했다.

**[ConcurrentHashMap: 사용자별 동시성 보장]**<br/>
포인트 데이터는 사용자별로 독립적이다. 즉, A 사용자에 대한 요청과 B 사용자에 대한 요청은 동시에 처리되어도 데이터 정합성에 문제가 없다.

만약 시스템 전체에 단 하나의 `ReentrantLock`을 사용하는 '글로벌 락(Global Lock)' 방식을 적용한다면, A 사용자의 작업이 끝날 때까지 B, C, D 사용자의 작업이 모두 불필요하게 대기해야 한다. 이는 동시 요청이 많아질수록 시스템 전체의 처리량을 심각하게 저하시킨다.

이러한 성능 저하를 방지하기 위해, **사용자 ID별로 개별 락을 관리**하는 방식을 채택했다.

- 구현 : `ConcurrentHashMap<Long, ReentrantLock>`
- 동작
    1. 요청이 들어오면 key(사용자 ID)로 `ConcurrentHashMap`을 조회한다.
    2. 해당 ID를 위한 `ReentrantLock` 객체로 락을 획득한다.
    3. 비즈니스 로직(포인트 차감/증가)을 처리한다.
    4. 락을 해제한다.

`ConcurrentHashMap`은 맵 자체의 스레드 안전성을 보장해 준다.

이 구조를 통해 **서로 다른 사용자에 대한 요청은 동시에 처리**하고, **동일한 사용자에 대한 요청(경쟁 상태가 발생하는 지점)만 순차적으로 처리**하여, 데이터 정합성과 시스템 전체 성능을 모두 확보했다.
